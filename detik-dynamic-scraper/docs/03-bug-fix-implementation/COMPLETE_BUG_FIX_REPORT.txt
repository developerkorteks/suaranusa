🔍 ROOT CAUSE ANALYSIS (100% COMPLETED)
================================================================================

Issue #1: Missing HTML Selectors
  Problem:
    - article_detail_scraper.py only had selectors for news.detik.com
    - Wolipop uses .itp_bodycontent class (different HTML structure)
    - Content extraction returned None for wolipop articles
  
  Evidence:
    - Tested wolipop URL → content = None
    - Tested news.detik.com URL → content = 2107 chars
    - HTML inspection showed .itp_bodycontent in wolipop pages

Issue #2: Unsafe None Handling in API
  Problem:
    - API endpoint: len(detail_data.get('content', ''))
    - When content exists but is None, .get() returns None (not '')
    - len(None) raises TypeError → HTTP 500
  
  Evidence:
    - Simulated API call → TypeError on line 346
    - .get('content', '') returns None if key exists with None value
    - No defensive None handling in response builder

================================================================================
✅ SOLUTION IMPLEMENTED
================================================================================

Fix #1: Enhanced Content Selectors
  File: src/core/article_detail_scraper.py
  Lines: 99-111
  
  Changes:
    BEFORE:
      selectors = [
          'article .detail__body-text',  # Only news.detik.com
          '.detail__body-text',
          'article',
          ...
      ]
    
    AFTER:
      selectors = [
          '.itp_bodycontent',      # NEW: Wolipop, Food, etc
          '.detail__body',         # NEW: General detik
          'article .detail__body-text',
          '.detail__body-text',
          '[itemprop="articleBody"]',
          '.article-content',
          '#article-content',
          '.entry-content',
          'article'  # Fallback
      ]
  
  Impact:
    ✅ Wolipop articles now extract content (2587 chars)
    ✅ Food, entertainment, and other subdomains supported
    ✅ Backward compatible with news.detik.com
    ✅ Comprehensive fallback chain

Fix #2: Safe Content Handling
  File: src/api/main.py
  Lines: 344-350
  
  Changes:
    BEFORE:
      return {
          "content_length": len(detail_data.get('content', '')),  # TypeError!
          "has_content": bool(detail_data.get('content')),
      }
    
    AFTER:
      content = detail_data.get('content') or ''  # Safe: always string
      
      return {
          "content_length": len(content),  # No TypeError!
          "has_content": bool(content),
      }
  
  Impact:
    ✅ No more TypeError crashes
    ✅ Graceful handling of None content
    ✅ Always returns valid HTTP 200 response
    ✅ Consistent API behavior

🎓 KEY LEARNINGS
================================================================================

1. Different Subdomains = Different HTML Structures
   - Lesson: Always test across all subdomains, not just one
   - Impact: Discovered wolipop uses .itp_bodycontent
   - Solution: Comprehensive selector list with priority ordering

2. .get(key, default) Doesn't Protect Against None Values
   - Lesson: If key exists with None value, .get() returns None
   - Impact: len(None) causes TypeError
   - Solution: Use 'or' operator: value.get('key') or 'default'

3. Defensive Programming Prevents Crashes
   - Lesson: Always handle None values explicitly
   - Impact: API crashes turned into graceful responses
   - Solution: Safe content handling pattern

4. Comprehensive Testing Catches Edge Cases
   - Lesson: Test multiple subdomains, not just one example
   - Impact: Found wolipop failure that wasn't obvious
   - Solution: Test suite with diverse URLs
